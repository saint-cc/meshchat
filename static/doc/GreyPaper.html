MeshChat: A Resilient Identity Messaging Protocol

Abstract

MeshChat is an experimental decentralized communication protocol focused on portable cryptographic identity, direct peer connectivity and resilient message delivery under unstable network conditions.

The system avoids centralized identity providers and minimizes infrastructure trust requirements. Identities are locally derived from user-held secrets and can migrate freely between devices, domains and relay nodes.

Communication prefers direct WebSocket connectivity between peers while preserving asynchronous delivery through relay-assisted fallback transport.

1. Introduction

Modern messaging systems typically bind identity, message storage and routing to centralized service providers. MeshChat explores an alternative model where identity is user-owned, reproducible and transport-independent.

The protocol operates entirely at the application layer and intentionally relies on existing Internet infrastructure rather than introducing custom network-layer routing systems.

MeshChat does not attempt to replace TCP/IP, DNS, BGP or existing Internet routing. Instead it defines a portable identity and message delivery layer operating above them.

The project prioritizes:

  • Portable cryptographic identity
  • Direct peer communication where possible
  • Graceful degradation under partial failure
  • Minimal infrastructure trust assumptions
  • Eventually consistent synchronization

2. Identity Model

Each identity is deterministically derived from a username and passphrase. No centralized registration authority exists.

Key material is locally generated using PBKDF2 and HKDF expansion:

  • Encryption key (AES-GCM)
  • Signing key seed (Ed25519)
  • Backup/recovery key

A stable public identifier (publicID) is derived from cryptographic material rather than infrastructure ownership.

Identity in MeshChat is cryptographic rather than account-based. A relay node does not create or own identities.

2.1 Identity Portability & Reconstruction

MeshChat identities are not tied to devices, sessions or domains. A user can reconstruct the same identity on any compatible client using only their credentials.

When a user appears on a new device:

  • The same cryptographic identity is recreated deterministically
  • The identity reconnects to known contacts as the same participant
  • Peer synchronization restores contact and message state progressively
  • Conversation history may be reconstructed through sync and encrypted backup exchange

Identity continuity exists independently from device continuity.

3. Communication Model

MeshChat uses a hybrid communication model combining direct peer transport and asynchronous fallback delivery.

3.1 Direct Peer Connectivity

Clients maintain a persistent WebSocket connection to a chosen home relay. However, messages are normally sent directly from one client to another client’s advertised WebSocket endpoint.

Alice → Bob.currentWSS

This allows low-latency communication without requiring relay-to-relay federation.

3.2 Home Relay Model

Each identity maintains a preferred relay association. The relay acts as:

  • Connection anchor
  • Fallback message transport
  • Temporary synchronization point
  • Reachability advertisement source

The relay is not a trust authority and cannot decrypt or validate encrypted payload contents.

3.3 Packet Structure

Messages are transmitted as opaque encrypted blobs:

  • Encrypted payload
  • Ed25519 signature
  • Sender publicID
  • Recipient publicID

Relays transport packets without access to message content.

4. Reachability & Migration

MeshChat identities may migrate between domains, devices or relay nodes without changing identity.

A contact advertisement may contain:

{ mail: "meshchat@domain.org", wss: "wss://domain.org/ws/" }

These fields serve different purposes:

  • wss → current preferred live endpoint
  • mail → durable fallback relay transport address

The current WebSocket endpoint may change frequently while the relay email may remain stable for extended periods.

Migration therefore becomes a gradual convergence process rather than a hard transition event.

Temporary divergence between known endpoints is tolerated by design.

5. Synchronization Model

MeshChat uses opportunistic peer synchronization rather than strict global consistency.

Peers may:

  • Exchange recent message windows
  • Compare synchronization state
  • Merge missing encrypted packets
  • Repair temporary divergence over time

The protocol prioritizes eventual consistency and graceful recovery over immediate consensus.

Temporary forks in local conversation state are expected and self-repair through synchronization.

6. Offline & Delay-Tolerant Delivery

If direct peer connectivity fails, messages may fall back to asynchronous relay-assisted delivery.

Relays may:

  • Temporarily queue encrypted packets
  • Forward packets through email transport
  • Retry delayed delivery
  • Expire stale packets using TTL policies

The client itself is transport-agnostic and does not directly interact with email systems. Fallback logic exists entirely within relay behavior.

Client → Home Relay → Email fallback → Recipient Relay → Recipient Client

7. Trust Model

MeshChat assumes infrastructure may be observable, unreliable or partially hostile.

Security assumptions:

  • Payloads remain end-to-end encrypted
  • Message authenticity is verified cryptographically
  • Relays may observe timing and routing metadata
  • Relays are not trusted for identity verification

Initial trust establishment is expected to occur out-of-band where possible:

  • QR exchange
  • In-person verification
  • Existing secure channels
  • TOFU-style onboarding

8. Relay Philosophy

Relays are intentionally minimal and non-authoritative.

Unlike federated systems, relays do not maintain global consensus or relay-to-relay routing protocols. Cross-domain communication occurs through ordinary Internet infrastructure and optional asynchronous transport methods.

A relay protects:

  • Its own availability
  • Its own resources
  • Its own transport integrity

A relay does not own:

  • User identity
  • Message authenticity
  • Conversation state truth

9. Design Philosophy

MeshChat explores communication as a resilient distributed state problem rather than a centralized service problem.

The system intentionally favors:

  • Identity portability
  • Graceful degradation
  • Transport flexibility
  • Local-first state ownership
  • Partial connectivity tolerance
  • Incremental self-repair

Failure is treated as a temporary synchronization condition rather than a terminal communication event.

10. Conclusion

MeshChat investigates a communication model where cryptographic identity remains stable while connectivity and infrastructure remain fluid.

The protocol separates identity from transport and allows communication to continue across changing domains, devices and partial network failures.

The project remains experimental and intentionally iterative. Future revisions may refine synchronization, migration handling and transport negotiation while preserving the core identity model.
